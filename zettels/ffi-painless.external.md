# FFI via Painless

Foreign Function Interface gated by [[painless.comparable]]'s security allowlist. `Java.call "class" "method" args`. Reuses Painless `WhitelistLoader` SPI. For quick prototyping, math/crypto/encoding, string manipulation.

Security model:
- Every `Java.call` is checked at [[elaboration-architecture.types]] time against the Painless allowlist (`WhitelistLoader` loads `.txt` allowlist files that enumerate permitted classes and methods)
- Disallowed calls fail at compile time with a clear error message naming the rejected class/method -- never at runtime
- Plugins can extend the allowlist via ServiceLoader (`PainlessExtension` SPI), so custom ES plugins can expose additional Java surface area to piescript via [[plugin-spi.external]] without modifying core allowlists

Open question: should piescript also check at runtime via ES's entitlement system for defense-in-depth? Elaboration-time checking is sufficient if the allowlist is correct, but a runtime guard would catch bugs in the allowlist loader or in dynamic code paths where the class/method name is not statically known (though the current design requires static string literals for `Java.call`, making this less of a concern).

The FFI bridges naturally to [[bytecode-compilation.performance]]. When piescript compiles pure fragments to JVM bytecode, FFI calls become direct `invokestatic` or `invokevirtual` method invocations with zero boxing overhead -- the Painless allowlist check happens once at compile time, and the generated bytecode calls the Java method directly.
